glucotracker project explainer
==============================

Last updated: 2026-05-01

Purpose
-------

glucotracker is a personal food diary for a type-1 diabetic. It logs meals,
macros, calories, photos, product labels, restaurant shortcuts, and optional
nutrients for journaling only.

It is not used for insulin dosing. The app must never recommend insulin, bolus,
correction, treatment, or medical decisions.

Core rule:

  Backend is the product.
  Frontend is replaceable.
  Backend owns the database, API semantics, application-service orchestration,
  macro math, draft lifecycle, Gemini normalization, product memory, daily
  totals, and Nightscout sync.
  The desktop UI is only one client implementation.


Stack
-----

Backend:

- Python 3.11+
- FastAPI
- SQLAlchemy 2.0 declarative models with Mapped[] annotations
- Alembic
- SQLite for local deployment, written through SQLAlchemy so moving to
  Postgres later stays straightforward
- Pydantic / pydantic-settings
- httpx
- pytest / pytest-asyncio
- ruff

Backend deployment:

- Docker Compose on Ubuntu server

Desktop client:

- Tauri 2
- React 18+
- TypeScript strict
- Tailwind
- TanStack Query
- Zustand
- react-router-dom
- generated TypeScript API client from OpenAPI

Future Android client:

- Separate Kotlin/Compose app
- Same REST + JSON backend
- No desktop-specific backend assumptions

Auth:

- Single bearer token from GLUCOTRACKER_TOKEN in backend .env
- No OAuth
- No users table

Secrets:

- Gemini API key lives only on backend
- Nightscout settings live only on backend
- Frontend must not contain Gemini or Nightscout secrets

Backend model overview
----------------------

Main tables:

- meals
- meal_items
- photos
- patterns
- pattern_aliases
- products

- nightscout_glucose_entries
- nightscout_insulin_events
- nightscout_import_state
- ai_runs / AI estimation run history
- nutrient_definitions
- meal_item_nutrients

Computed report data:

- GET /reports/endocrinologist returns presentation-ready JSON for the
  endocrinologist PDF.
- It is computed on demand from accepted meals and local Nightscout context.
- It is not persisted as PDF-specific rows.
- Food episodes, linked insulin, observed ratio, TIR, glucose windows, daily
  summaries, warnings, and footer safety text are backend-owned.
- Frontend renders and saves the PDF only.

Important meal statuses:

- draft: editable review state
- accepted: journal entry counted in daily totals
- discarded: hidden/cancelled draft or removed workflow state

Important meal sources:

- photo
- pattern
- manual
- mixed

Important item sources:

- photo_estimate
- label_calc
- restaurant_db
- product_db
- pattern
- manual

Unknown optional nutrients are null, not 0. Manual overrides have the highest
priority. Label/product/restaurant database values beat Gemini visual estimates.
Gemini can extract visible facts, but backend performs deterministic
calculation.


API and lifecycle
-----------------

Public endpoints:

- GET /health
- GET /openapi.json

Protected endpoints require:

  Authorization: Bearer <GLUCOTRACKER_TOKEN>

Core meal lifecycle:

1. Create a meal as draft or accepted.
2. Add manual items, pattern/product items, or upload photos.
3. For photos, call estimate or estimate_and_save_draft.
4. Gemini extracts label/visual information.
5. Backend normalizes suggestions and calculates totals.
6. UI displays draft items, evidence, assumptions, warnings, confidence.
7. User edits draft.
8. UI sends reviewed items to POST /meals/{id}/accept.
9. Backend replaces items atomically, recalculates totals, and sets accepted.
   For label_calc items, backend recalculates label totals from extracted
   facts/evidence before persistence instead of trusting submitted item totals.
10. Accepted label_calc products may be remembered into the local product DB.
11. Accepted meals count in dashboard/daily totals.
12. Drafts do not count in daily totals.

UX rules
--------

Meal rows:

- show time on the left
- show thumbnail when available
- show title
- show source/status subtitle
- show dotted spacer
- show carbs/protein/fat/kcal aligned to the right
- show draft badge for drafts
- show quantity badge, for example x2, when applicable
- row totals reflect what was eaten, not per-unit values

Drafts:

- draft rows stay editable after click-away
- selecting a draft reopens draft review mode
- draft can be saved, edited, or discarded
- clicking away does not accept or delete
- draft does not count in daily totals

Images:

- uploaded photo has priority for photo meals
- meal.thumbnail_url is preferred for row thumbnail
- item image fields are fallback
- broken image shows placeholder
- database product images can be replaced from the database right panel

Autocomplete:

- plain text searches across saved products, personal patterns, restaurant
  patterns, aliases, display names, brand, key/token
- prefix search still narrows namespace, for example bk: or my:
- saved personal/product results should rank above restaurant DB for plain
  search
- selecting saved product logs product_db item without Gemini

Time handling:

- desktop sends local wall-clock datetime without trailing Z
- if user enters 12:30, the journal should save and display 12:30
- local day filters also use local datetime boundaries
- creating food from the Journal uses the selected day, not always today
- if user has 28 April open and adds a photo/manual meal, the meal is created
  on 28 April with current local time
- selected meal panel must allow editing full date/time, not only HH:MM
- PATCH /meals/{id} with eaten_at moves the meal and backend recalculates old
  and new daily totals

Repeat-by-weight:

- selected meal panel can create a new accepted meal from one existing item
- frontend calls POST /meal_items/{id}/copy_by_weight with target grams and
  intended local eaten_at
- backend scales macros/nutrients, copies source metadata/photo reference, and
  returns the new accepted meal
- for label items with count and per-unit weight, UI should show quick repeat
  for one recognized unit/package
- example: accepted halva item "x3 packages, 20 g each" has grams=60 and
  evidence count=3; quick action "Добавить 1 упаковку" sends grams=20
- frontend must not calculate accepted repeat totals locally

Nightscout timeline:

- Nightscout insulin is read-only context imported from Nightscout.
- Insulin context is shown as a manual Nightscout event, not as a local meal
  item and never as dosing advice.
- Food remains a stream of meal rows. The backend computes food episodes for
  history display; it does not merge or replace the underlying meals.
- Episode grouping rule: consecutive accepted meals within 30 minutes belong to
  the same food episode.
- Episode context rule: insulin is linked from 30 minutes before the first meal
  to 90 minutes after the last meal.
- CGM context rule: local Nightscout glucose points are linked from 60 minutes
  before the first meal to 180 minutes after the last meal.
- The desktop history screen should render the episode as one visual block with
  nested food rows, insulin context rows, and a small CGM graph.

Endocrinologist report:

- The report is informational only.
- The label is "НАБЛЮДАЕМЫЙ УК" / "Наблюдаемый УК", never "recommended".
- Observed UK uses grams of carbs per registered meal-linked insulin unit:
  Σ carbs / Σ meal-linked insulin.
- Unlinked insulin/corrections count in daily total insulin but must not
  distort breakfast/lunch/dinner observed ratio.
- The desktop Settings page chooses date range, imports Nightscout context if
  enabled, calls GET /reports/endocrinologist, renders the A4 PDF with
  @react-pdf/renderer, and saves it through Tauri.
- Do not duplicate report aggregation math in React components.


Photo and Gemini estimation pipeline
------------------------------------

High-level flow:

1. User attaches photos by file picker, drag/drop, or clipboard.
2. UI keeps photos in pending state and shows thumbnails in right panel.
3. User can add optional context, for example "100 г вареного риса".
4. User clicks Оценить.
5. UI creates a draft photo meal with local eaten_at.
6. UI uploads photos to backend.
7. Backend stores files under data/photos/YYYY/MM/{photo_id}.{ext}.
8. Backend calls Gemini with:
   - system prompt
   - optional user context
   - photo manifest with photo ids, filenames, and order
   - selected patterns/products context where applicable
   - image bytes in the same order as the manifest
9. Gemini returns structured EstimationResult.
10. Backend stores raw response in ai_runs and on first photo.
11. Backend normalizes Gemini output into meal items.
12. Backend calculates all deterministic totals.
13. Backend may create one or many draft journal rows:
    - unrelated photos/items become separate draft rows
    - identical packages with split label evidence can remain one row with count
14. UI shows draft review cards with photos, evidence, assumptions,
    confidence, calculation details, source model, and totals.
15. User edits and accepts or discards.

Multiple photos:

- Each photo has photo_id, upload order, filename, and file URL.
- Gemini must return source_photo_ids/source_photo_indices for each item.
- Drink photo + lavash photo should become two draft rows.
- Two identical candies can become one row with count_detected = 2.
- If multiple photos are uploaded but only one object is found, backend/UI
  warns the user to check the result.

Label photos:

- LABEL_FULL: label facts and weight/volume visible.
- LABEL_PARTIAL: label facts visible but weight/volume missing.
- SPLIT_LABEL_IDENTICAL_ITEMS: nutrition facts visible on one identical package
  and net weight visible on another.
- Gemini extracts visible facts.
- Backend calculates totals from per-100g/per-100ml/per-serving values.
- Backend can create label_calc item.

Plated food:

- Gemini estimates components and macros.
- Backend can use saved known components to replace model component values.
- Example: if tortilla 40g is saved as product/component, backend uses its
  known carbs/protein/fat/fiber/kcal instead of Gemini tortilla estimate.
- Backend avoids double counting by replacing the raw component estimate, not
  adding known values on top of full model total.
- Known-component matching must be specific. Generic overlap such as "base",
  "component", "osnova", or "komponent" must not connect unrelated foods such
  as cake base and tortilla/lavash.

Whole fruit on scale:

- If an unpeeled orange/mandarin is seen on a scale, visible scale weight is
  treated as gross weight including peel unless user context says it is peeled
  or edible weight.
- Backend applies edible-yield correction for supported fruit.
- Example: orange 210g gross -> about 155.4g edible -> about 18.3g carbs.
- Raw Gemini estimate remains in evidence.
- Warning is stored that weight likely includes peel.

Product memory:

- Accepted label_calc items can be remembered into the local product database.
- Saved products get aliases.
- Later autocomplete can find them by plain text or product/my prefix.
- Selecting product logs immediately from product_db without Gemini.

Re-estimation:

- Existing photo meal can be re-estimated with another model.
- Re-estimate creates a comparison proposal.
- It does not overwrite current meal until user applies it.
- User can replace current, save as draft, or keep current.
- AI run history stores model, result, errors, normalized items, and promotion.


Gemini model routing and retry
------------------------------

Environment variables:

- GEMINI_MODEL
- GEMINI_FALLBACK_MODELS
- GEMINI_CHEAP_MODEL
- GEMINI_FREE_TEST_MODEL
- GEMINI_FALLBACK_MODEL
- GEMINI_MAX_RETRIES_PER_MODEL

Current intended default:

- GEMINI_MODEL=gemini-3-flash-preview
- fallback examples: gemini-2.5-flash, gemini-3.1-flash-lite-preview

Routing:

- explicit model override from UI can be used for estimate/re-estimate
- default route uses backend settings
- cheap model may be used for simple label extraction or quota-saving modes
- Pro models are not used automatically

Error handling:

- 503 UNAVAILABLE:
  retry with bounded backoff, then fallback model
- 429 RESOURCE_EXHAUSTED:
  do not spam retries; report quota/rate-limit state
- 400 invalid request:
  do not retry as normal user failure
- JSON/schema parse failure:
  retry once or fallback

AI run metadata:

- model_requested
- model_used
- fallback_used
- attempts
- error_history
- latency_ms


Current Gemini prompt
---------------------

The source of truth is:

  backend/glucotracker/infra/gemini/client.py
  PHOTO_ESTIMATION_PROMPT_V1

Clean readable version of the current prompt intent:

  You are a nutrition estimator for a personal food diary. The user is a
  type-1 diabetic who logs meals to track macros, but this is NOT used for
  insulin dosing. Do not recommend insulin, bolus, correction, or treatment.

  Input: one or more photos of one meal/snack. The user may attach multiple
  photos. Each photo is identified in the PHOTO MANIFEST and image parts are
  supplied in the same order.

  The request may include USER CONTEXT supplied by the user, such as known
  component weights, eaten quantity, or corrections like "100 г вареного риса".
  Use this as user-provided evidence. Do not ignore visible photo evidence. If
  it gives a known weight/count for a visible component, apply that context and
  explain it in evidence or assumptions. Do not use user context for insulin or
  medical treatment advice.

  For each distinct visible item:

  1. Identify the item and brand if recognizable.
  2. Choose scenario:
     - LABEL_FULL: nutrition facts and net weight/volume visible.
     - LABEL_PARTIAL: nutrition facts visible but net weight/volume missing.
     - PLATED: prepared food without label.
     - BARCODE: barcode visible.
     - SPLIT_LABEL_IDENTICAL_ITEMS: facts split across clearly matching
       identical packages/photos.
     - UNKNOWN: unclear.
  3. For labels, extract facts exactly. Do not do final arithmetic. Put visible
     Russian text into visible_label_facts and normalized per-100g values into
     nutrition_per_100g when visible. Extract optional visible nutrients such as
     fiber, sodium, caffeine, sugar, potassium, iron, calcium, magnesium. The
     backend performs final arithmetic.
  4. For missing weight/volume, provide assumption and reason.
  5. For plated food, estimate grams and macros low/mid/high. Do not visually
     guess sodium or caffeine. Fiber may be included only from known food
     database/context and must be marked estimated.
  6. Return component_estimates for plated food with component-level raw macros:
     carbs, protein, fat, fiber, kcal. Identify reusable known components such
     as tortilla/lavash, bread, rice, pasta, potato, sweet drink, candy/bar,
     cereal/granola, bakery items, and branded/standard components. Mark them
     should_use_database_if_available=true.
  7. If a reference object exists, use it for rough scale.
  8. Return confidence 0..1 and a one-sentence confidence reason.
  9. Put evidence as short user-readable strings, preferably Russian when label
     is Russian.
  10. For every item, set source_photo_ids, source_photo_indices, and
      primary_photo_id.

  Do not merge unrelated photos. If multiple photos show different food/drink
  items, return separate EstimatedItem objects. Do not merge a drink and a wrap
  into one item. Only combine evidence across photos when photos clearly show
  the same packaged product or identical packages.

  For one coherent wrap/lavash/sandwich on one photo, return one EstimatedItem
  with component_estimates, not separate final items for tortilla, chicken,
  tomatoes, sauce, etc. Component breakdown belongs inside the item.

  Component types:

  - carb_base: tortilla/lavash/bread/rice/pasta/potato/sweet drink/candy/
    cereal/bakery.
  - protein: chicken/meat/fish/egg/tofu.
  - vegetable: vegetables/greens.
  - sauce: sauces/dressings.
  - fat_source: oil/butter/cheese/nuts/avocado when primarily fat.
  - known_component: any visible component that looks like a saved product or
    pattern.
  - unknown: unclear component.

  Do not double count component macros. EstimatedItem mid totals should be the
  sum of visible components. Component mid macros should represent each
  component only, so backend can replace one component's raw values with saved
  database values.

  Russian output:

  - Return user-facing names in Russian.
  - Use display_name_ru for UI.
  - Do not return English names like "Chicken and Vegetable Wrap" unless no
    Russian identification is possible.
  - Prefer names such as:
      "Лаваш с курицей"
      "Ролл с курицей"
      "Напиток"
      "Шоколадный батончик"
      "Бисквит-сэндвич"

  Russian label mapping:

  - углеводы = carbs
  - белки = protein
  - жиры = fat
  - клетчатка / пищевые волокна = fiber
  - сахара = sugar
  - соль = salt
  - натрий = sodium
  - кофеин = caffeine
  - энергетическая ценность = kcal / energy
  - масса нетто = net weight
  - пищевая ценность на 100 г = nutrition per 100g

  Split label evidence:

  If multiple identical packaged items are visible and nutrition values are
  visible on one package while net weight is visible on another, combine those
  visible facts only when packages clearly appear identical. Keep scenario
  LABEL_FULL, set evidence_is_split_across_identical_items=true, return count,
  confidence, net weight per unit, total weight if directly derivable from
  visible count, nutrition_per_100g, and evidence explaining which facts came
  from which package. Do not calculate final macro totals; backend does that.

  Count:

  If multiple identical packages are visible, return count_detected. If
  uncertain, lower count_confidence and add an assumption. Do not assume eaten
  count is more than visible count.

  Whole fruit on scales:

  If unpeeled fruit such as orange, mandarin, banana, or similar whole fruit is
  visible on a scale, treat visible scale weight as gross weight including peel
  unless user context says it is peeled/edible weight. Explain this and lower
  confidence if edible yield is uncertain. Do not report high confidence for
  edible macros when only gross unpeeled weight is known.

  Confidence rubric:

  - >0.85: LABEL_FULL or strong product/pattern match.
  - 0.60..0.85: LABEL_PARTIAL with plausible size assumption, or clear plated
    item.
  - <0.60: mixed plated food, unclear photo, no reference, ambiguous item.

  Be conservative. Prefer uncertainty over fake precision.

Important safety checks
-----------------------

- No insulin advice.
- No frontend Gemini key.
- No Nightscout secrets in frontend.
- Draft totals do not count in accepted daily totals.
- Unknown nutrients are null, not zero.
- Sodium and caffeine are not visually guessed from plated food.
- Backend recalculates all accepted totals.
- Accepting label_calc items recalculates from label evidence/extracted facts,
  not from client-submitted total macros.
- Known-component matching is conservative; generic token overlap is ignored.
- Frontend uses generated API types and does not own final macro math.
- Multi-photo estimation preserves photo identity.
- Unrelated photos create separate draft journal rows.
- Same-product split-label evidence can become one counted row.
- Product DB selection does not call Gemini.
- Local wall-clock time remains local, not shifted by UTC conversion.
