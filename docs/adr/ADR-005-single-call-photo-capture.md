# ADR-005 · Single-call photo capture with server-side estimation

> Companion to ADR-001 / ADR-002 / ADR-003 / ADR-004. This is the most consequential of the ADRs so far: it dramatically simplifies the photo capture pipeline by collapsing five client-driven HTTP calls into one and moving Gemini orchestration to the server. Hand all five ADRs to the agent alongside the original prompt files.

## 1 · Context

`android-photo-estimate-flow.txt` documents the current capture pipeline. One user shutter-press produces five client-driven HTTP calls in sequence:

1. `GET /meals?from=…&to=…` — idempotency search for an existing draft within ±36h
2. `POST /meals` — create a draft meal shell
3. `POST /meals/{id}/photos` — multipart upload of the JPEG
4. `POST /meals/{id}/estimate_and_save_draft` — Gemini estimation + saving items
5. `GET /meals/{id}` — fetch the now-accepted meal for client display

Each step has its own failure mode, retry policy, and state-reconciliation logic. The Android `OutboxItem.state` machine (`Queued → Uploading → Estimating → Confirmed`) is a thin reflection of the server-side process, but the server is **not** the source of truth for the orchestration — the Android worker is.

Field evidence (screenshots from 2026-05-10 08:18 with shutter at 07:17, 1 hour later):

- Two `оценивается` rows on Today; banner says `1 в очереди`; queue inspector shows 1 active item. The depth is wrong somewhere — UI says 2 pending, queue logic says 1.
- The 1-hour-pending row has not aged into ADR-003's warning-tone state. Either ADR-003 §2.4 isn't implemented, or the state-entry timestamp isn't being read correctly.
- The flow document §15 already admits a similar class of bug: "the backend was already doing useful work… the Android bug was after the successful estimate response… Old Android required current.status == ACCEPTED. That was too strict." A patch trusts `created_drafts[0].meal_id` instead. This is treating one symptom of the deeper disease.

The disease is the architecture: **one user action is decomposed into five client-managed RPCs, each with its own race conditions, each requiring distinct recovery logic, and the client is responsible for keeping a state machine consistent with the server's actual progress without a reliable signal from the server.**

ADR-005 collapses this. The server takes ownership of orchestration. The client makes one call, gets back a meal id, and waits for a completion signal.

## 2 · Decisions

### 2.1 — One endpoint replaces five

```
POST /v1/meals/from-photo
Content-Type: multipart/form-data

  photo            (required, JPEG bytes, ≤10 MB after server's own resize)
  captured_at      (required, ISO 8601 with timezone)
  context          (optional, ≤500 chars, per ADR-004)
  source           (required, "camera" | "gallery")
  idempotency_key  (required, UUID generated by client at outbox enqueue)

→ 202 Accepted
  {
    "meal_id":          "uuid",
    "estimate_status":  "estimating",
    "captured_at":      "2026-05-10T07:17:00.000+03:00",
    "photo_url":        "/v1/photos/{uuid}/file"
  }
```

The server, on this single request:

1. Validates auth, content type, size limits, idempotency key.
2. If a meal with this `idempotency_key` already exists for this user, returns it (status reflects current state — could be `estimating`, `succeeded`, `failed`, `timeout`).
3. Otherwise, creates the meal record (status `estimating`), stores the JPEG, returns 202 within ~500 ms.
4. **Asynchronously**, server kicks Gemini estimation. Gemini call has a hard 90-second timeout. Outcome updates the meal's `estimate_status`.
5. Server fires an FCM data-message to the user's registered devices when status transitions out of `estimating`.

The five client RPCs from `android-photo-estimate-flow.txt` are gone. Client makes one call. Server owns the rest.

### 2.2 — Server-side terminal estimate states

`meal.estimate_status` enum:

| Status | Meaning |
|---|---|
| `estimating` | Gemini call in progress; client renders `оценивается` |
| `succeeded` | Gemini returned items; meal has nutrition values |
| `failed` | Gemini returned, but no recognizable food (clear photo, no items) |
| `timeout` | Gemini didn't respond within 90 seconds |
| `error` | Gemini returned an error (auth, quota, malformed response) |

`failed` / `timeout` / `error` are terminal-but-recoverable. The client surfaces them with explicit copy and a `Переоценить` action (the same endpoint as ADR-004 §2.4).

`estimating` is bounded. If 90s elapse without Gemini responding, server-side scheduler transitions the meal to `timeout` automatically. **There is no scenario where a meal sits in `estimating` for an hour.** That state of affairs from the screenshots becomes structurally impossible.

### 2.3 — Client outbox state machine collapses

Per ADR-001, the client's `OutboxItem.state` was: `Queued → Uploading → Estimating → Confirmed | Stuck`. Under ADR-005, the client no longer tracks the server-side estimation. State machine collapses to:

| State | Meaning |
|---|---|
| `Queued` | enqueued locally, not yet uploaded |
| `Uploading` | multipart upload in flight |
| `Confirmed` | server returned 202 with `meal_id` |
| `Stuck` | upload exhausted retries (5 attempts) or 4xx response |

Note: `Estimating` is removed from the **outbox** state machine. The client's outbox finishes its job once it has a `meal_id`. The estimate status (`estimating | succeeded | failed | timeout | error`) is a property of the **meal record**, not the outbox item, and it's read directly from the cached `meals` table.

The journal row's pending sub-states still exist, but they now read from `meals.estimate_status`, not from `OutboxItem.state`:

| Row state | UI |
|---|---|
| outbox is `Queued` or `Uploading`, no meal yet | `ждёт сети` (in numbers area) |
| meal exists, status `estimating` | `оценивается ···` |
| meal exists, status `succeeded` | numeric values |
| meal exists, status `failed` | `модель ничего не нашла · повторить` (warn tone) |
| meal exists, status `timeout` | `оценка не пришла за 90 секунд · повторить` (warn tone) |
| meal exists, status `error` | `ошибка оценки · повторить` (warn tone) |

The 10-minute aging warning from ADR-003 §2.4 is **removed**. It's no longer needed: the server provides the terminal state directly. If `estimating` lasts >90s server-side, the meal becomes `timeout` and the row goes warn-tone immediately — no client-side timer.

### 2.4 — Completion signal: FCM push, with foreground polling fallback

When the server transitions a meal out of `estimating`, it fires an FCM data-message:

```json
{
  "type": "meal_estimate_completed",
  "meal_id": "<uuid>",
  "estimate_status": "succeeded"
}
```

Client receives it (silent, no notification UI), refreshes the meal via `GET /v1/meals/{id}`, updates the cache, and the journal row animates per ADR-001 §3.

If FCM is unavailable (no Google Play Services, custom ROM, etc.):

- **Foreground**: while Today is the active screen and the user has any meal in `estimating` state, client polls `GET /v1/meals/{id}` every 5 seconds until the status changes. Stops polling when meal becomes terminal or screen leaves foreground.
- **Background**: client refreshes on next app foreground via the existing dashboard fetch.
- **Periodic**: a 15-min `PeriodicWorkRequest` continues to exist as a final safety net, refreshing any meals known to be `estimating` from the local cache.

This three-layer fallback (FCM → foreground polling → periodic refresh) means estimate completion always reaches the user, with FCM giving near-instant feedback when available.

### 2.5 — Idempotency is structural

The client generates a UUID at outbox enqueue (`idempotency_key`). The server, on receiving `POST /v1/meals/from-photo`:

- If a meal exists for this user with this `idempotency_key` → return its current state. Don't re-upload, don't re-estimate.
- Otherwise create a new meal with this key.

`idempotency_key` is a unique-per-user index server-side. The 36-hour `GET /meals` search from the old flow is **deleted**. Idempotency is no longer a heuristic over timestamps; it's a primary key.

This also means the client can safely retry indefinitely on transient failures — duplicate uploads can't create duplicate meals.

### 2.6 — Explicit retry, not auto-retry, for terminal estimate failures

A meal in `failed`, `timeout`, or `error` state does **not** auto-retry. The client surfaces a `Повторить` action on the row and on the queue inspector. Tap → calls `POST /v1/meals/{id}/re-estimate` (the same endpoint added in ADR-004 §2.4) → server kicks a new Gemini call → status returns to `estimating` → cycle repeats.

This is by design: a `timeout` likely means Gemini was slow once and might be fine on retry; an `error` likely means a quota/auth issue and the user should check the queue inspector. Either way, silently retrying without user awareness is exactly what ADR-003 §2.1 forbade.

## 3 · Specifications

### 3.1 — Backend: new schema additions

`meals` table:

```sql
ALTER TABLE meals ADD COLUMN estimate_status TEXT NOT NULL DEFAULT 'succeeded';
  -- enum: estimating | succeeded | failed | timeout | error
ALTER TABLE meals ADD COLUMN estimate_started_at TIMESTAMPTZ;
ALTER TABLE meals ADD COLUMN estimate_finished_at TIMESTAMPTZ;
ALTER TABLE meals ADD COLUMN estimate_error_code TEXT;       -- nullable, free-form
ALTER TABLE meals ADD COLUMN estimate_error_message TEXT;    -- nullable, user-safe Russian
ALTER TABLE meals ADD COLUMN idempotency_key UUID UNIQUE;    -- per-user; partial unique index on (user_id, idempotency_key)
```

Backfill for existing rows: `estimate_status = 'succeeded'` (existing meals already have values), other fields NULL. Forward-only migration.

Per-user FCM tokens: a new `device_tokens` table:

```sql
CREATE TABLE device_tokens (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id),
  fcm_token TEXT NOT NULL,
  device_label TEXT,
  created_at TIMESTAMPTZ NOT NULL,
  last_seen_at TIMESTAMPTZ NOT NULL,
  UNIQUE (user_id, fcm_token)
);
```

Token registration: `POST /v1/device-tokens` from the client at app launch, on auth, and on FCM token refresh.

### 3.2 — Backend: estimation worker

The synchronous `POST /meals/{id}/estimate_and_save_draft` from the old flow is rewritten as an asynchronous worker job (Celery, ARQ, or FastAPI background task — engineer's choice; Celery preferred for robustness):

```
Worker: estimate_meal(meal_id)
  1. Load meal, verify estimate_status == 'estimating'
  2. Set estimate_started_at = now
  3. Call Gemini with timeout=90s, photos + context
  4. On success: parse, save items, update totals, set estimate_status='succeeded'
  5. On no-items response: set estimate_status='failed'
  6. On timeout: set estimate_status='timeout'
  7. On other Gemini errors: set estimate_status='error', store error_code/message
  8. Set estimate_finished_at = now
  9. Fire FCM to all of the meal's user's device_tokens
```

The worker is idempotent: if a meal is already in a terminal state, the worker returns immediately. (This handles the edge case where the user retries before the original Gemini call returns.)

### 3.3 — Backend: deprecate but don't remove old endpoints

For the migration window:

- `POST /v1/meals/from-photo` is the new canonical entry point.
- `POST /v1/meals` (creating meal shell), `POST /v1/meals/{id}/photos` (uploading photo), `POST /v1/meals/{id}/estimate_and_save_draft` (Gemini call) all continue to work for the 4 weeks following deployment, then are removed.
- Internally, the new endpoint can call the same underlying functions, just orchestrated server-side. No need to rewrite the Gemini integration; just rewrite the orchestration layer.

### 3.4 — Client: outbox state simplification

`OutboxKind.CapturedMeal` payload gains `idempotencyKey: UUID` (generated at enqueue, persisted in the outbox row).

`OutboxState` enum loses `Estimating`. Only `Queued | Uploading | Confirmed | Stuck`.

`OutboxProcessorImpl.processCapturedMeal` becomes:

```kotlin
suspend fun processCapturedMeal(item: OutboxItem) {
  markUploading(item.id)
  val response = mealsApi.fromPhoto(
    photo = photoStorage.bytes(item.localPhotoPath),
    capturedAt = item.capturedAt,
    context = item.photoContext,
    source = item.source,
    idempotencyKey = item.idempotencyKey,
  )
  mealsCache.upsert(response.toMeal())   // status = estimating
  markConfirmed(item.id, response.mealId)
}
```

If the request fails: standard 5-attempt exp backoff per ADR-003 §2.1, then `Stuck`. No special handling for the estimating phase — that's not the outbox's job anymore.

### 3.5 — Client: FCM integration

Add `firebase-messaging` dependency, gated to the gluco and food flavors that ship through Google Play (i.e., not behind a flag — both flavors get it). Self-distributed APKs get the same code; FCM either works or it doesn't, and the polling fallback covers the latter case.

`GTFirebaseMessagingService` handles the data-message:

```kotlin
override fun onMessageReceived(message: RemoteMessage) {
  if (message.data["type"] == "meal_estimate_completed") {
    val mealId = UUID.fromString(message.data["meal_id"])
    workScheduler.enqueueMealRefresh(mealId)
  }
}
```

`MealRefreshWorker`: hits `GET /v1/meals/{mealId}`, updates `cached_meals`, observers (Today's `Flow`) pick up the change.

Token registration: `MainActivity.onCreate` and `AuthRepository.onLogin`/`onTokenRefresh` register/refresh the FCM token via `POST /v1/device-tokens`.

### 3.6 — Client: foreground polling fallback

`TodayViewModel` exposes a derived `Flow<List<UUID>>` of meals currently in `estimate_status = estimating`. When that flow has at least one element AND the screen is in foreground (lifecycle observer), a coroutine polls `GET /v1/meals/{id}` every 5 seconds for each.

Polling stops when:
- the screen leaves foreground,
- the list becomes empty,
- a poll returns a terminal status (the corresponding meal_id is removed from the list).

Each poll is a single GET, no batching needed for the typical 0–2 in-flight estimates. If the user does a "burst capture" of 10 photos, the polling cost is 10 GETs every 5s × 90s ≈ 180 requests in the worst case before all timeout — acceptable.

## 4 · Section overrides

### 4.1 Supersedes most of `T6 · Capture flow` (post ADR-001)

ADR-001 collapsed the two-step `PhotoEstimateRequest + AcceptDraft` into one `CapturedMeal`. ADR-005 simplifies further:

- The 5-RPC pipeline from `android-photo-estimate-flow.txt` §8–11 is **deleted**. Replaced by §3.4 (one call).
- `KtorOutboxRemote.captureMeal` is rewritten end-to-end. The 36-hour idempotency search via `GET /meals?from=...` is gone — replaced by structural idempotency_key (§2.5).
- `findExistingPhotoMeal` helper from the old `KtorOutboxRemote` is **deleted**.
- `EstimateMealRequest` / `EstimateMealResponse` / `EstimateCreatedDraftResponse` schemas are **deleted** from the OpenAPI spec after migration.

### 4.2 Supersedes parts of `T3 · Outbox + sync infrastructure`

- `OutboxState.Estimating` is **removed**. The outbox machine is `Queued | Uploading | Confirmed | Stuck`.
- The 10-min `Estimating` timeout from ADR-003 §2.1 is **removed** — server owns this now (90s timeout there).
- `outbox_items` table loses any column that tracked client-side estimating progress.

### 4.3 Supersedes parts of `ADR-001 · Synchronous capture, deferred estimate`

- ADR-001 §2.1 ("State machine: `Queued | Uploading | Estimating | Confirmed | Stuck`") becomes `Queued | Uploading | Confirmed | Stuck` (estimating removed).
- ADR-001 §2.3 (Status taxonomy) is updated: the `Pending · estimating` row state now reads from `meals.estimate_status`, not from outbox state.
- ADR-001 §3 (animations) unchanged. The arrival animation now fires when `meals.estimate_status` transitions to `succeeded`, regardless of how that transition arrived (FCM push, polling, periodic refresh).

### 4.4 Supersedes parts of `ADR-003 · Outbox state visibility`

- Banner state matrix from ADR-003 §2.2 is unchanged in structure but the input changes: "active queue" now counts only outbox items in `Queued` or `Uploading`. Estimating-but-server-confirmed meals don't count toward the banner — they're confirmed work.
- The "stuck queue" includes outbox-`Stuck` items as before, plus meals with terminal-bad estimate states (`failed`, `timeout`, `error`). The banner copy is unchanged: `N не отправилось · посмотреть`.
- The queue inspector (ADR-003 §2.3) shows two sources: outbox items and meals with bad estimate status. UI distinguishes them inline (e.g., "не загрузилось" for outbox-Stuck vs "оценка не пришла за 90 секунд" for meal-timeout).

### 4.5 Compatible with `ADR-002 · Capture compose unification` and `ADR-004 · Photo context input`

- ADR-002 compose sheet path → camera → shutter is unchanged at the UI level. Under the hood, the camera flow now enqueues a `CapturedMeal` whose worker hits `/meals/from-photo` (one call). The compose sheet's text-as-context behavior from ADR-004 §3.3 plumbs through unchanged.
- ADR-004's `photo_context` is now a field in the `POST /v1/meals/from-photo` multipart request. The server stores it on the meal and includes it in the Gemini prompt (per ADR-004 §2.3).
- The re-estimate endpoint (`POST /v1/meals/{id}/re-estimate`) from ADR-004 §2.4 is now also the recovery action for `failed`/`timeout`/`error` states from §2.6 here.

### 4.6 `android-photo-estimate-flow.txt` is fully superseded

The 18-section flow document describes the old 5-RPC pipeline. After ADR-005 ships, that document is replaced by a much shorter one (§5 below produces it).

## 5 · Implementation tasks (phased)

- **A3**. Background worker for estimation per §3.2. Update the meal row when Gemini returns or times out.

- **A7**. Tests: idempotency_key dedup; concurrent uploads with same key; Gemini timeout produces `timeout` state; FCM fires on completion.

### Phase B · Android (1 week)

- **B2**. `GTFirebaseMessagingService` per §3.5. Token registration on auth/launch/refresh.

- **B4**. `OutboxProcessorImpl.processCapturedMeal` rewritten per §3.4. One call. Old multi-step `KtorOutboxRemote.captureMeal` is replaced wholesale; old findExistingPhotoMeal/createDraftMeal/uploadPhoto/estimateAndSave functions deleted.

## 6 · Acceptance

- **No more 1-hour `оценивается`.** A photo captured with no network → comes online 5 minutes later → uploads via the new endpoint → server estimates in ≤90s → FCM fires → row shows numeric values within ~6 minutes total. End-to-end test on a real device.
- **State consistency.** `Today` row count of pending entries always matches the queue inspector's "active" count. Banner depth always matches sum of (outbox-`Queued`/`Uploading`) + (meals-`estimating`). Compose UI test asserts equality across all three surfaces in 10 random states.
- **Idempotency under retry.** Capture a photo, force-kill the app mid-upload, restart. Worker retries with the same `idempotency_key`. Server returns the existing meal. No duplicate row in `meals`.
- **Server-side timeout.** Mock Gemini to never respond. After 90s, meal `estimate_status` becomes `timeout`. Client journal row shows `оценка не пришла за 90 секунд · повторить` in warn tone.
- **Recovery via re-estimate.** Tap `Повторить` on a `timeout` meal → `POST /re-estimate` fires → status returns to `estimating` → Gemini responds normally → values populate. Whole cycle ≤2 min.
- **FCM delivery.** With FCM working, a successful estimate produces a row update in the journal within ~2 seconds of server completion. Without FCM, foreground polling picks up the change within 5 seconds. Background-without-FCM picks up on next app foreground.
- **No client-side estimating timer.** Codebase grep shows no client logic that watches "time since outbox state entered `Estimating`". This is server's job now.
- **Documentation.** `docs/photo-capture.md` (replacing `android-photo-estimate-flow.txt`) describes the 1-call flow in ≤4 sections, ≤80 lines.

## 7 · Risks and mitigations

- **FCM dependency.** Adds a Google Services dependency. Mitigation: foreground + periodic polling fallback (§3.6) makes FCM strictly an optimization, not a requirement. Apps work without FCM, just with up to 5-second latency on foreground or refresh-on-open delay otherwise.
- **Backend complexity moves.** Server now hosts the estimation orchestration that used to live on the client. Server worker queue must be reliable. Mitigation: use a battle-tested queue (Celery + Redis or ARQ + Redis); make the worker job idempotent (§3.2); monitor queue depth.
- **Gemini quota cost may rise** if re-estimate is too easy to trigger. Mitigation: rate-limit per ADR-004 §7.3 (max 1 re-estimate per meal per 30s); no daily cap initially; add monitoring before any scaling.
- **Migration window state confusion.** Some users on the new client, others on the old. Mitigation: the new endpoint works on its own; the old endpoints continue to serve the old client code; no cross-contamination. Phase D doesn't start until rollout reaches >95%.

## 8 · Out-of-band asks

1. **FCM project credentials.** Backend needs Firebase Admin SDK service account. Confirm that the user is willing to set up a Firebase project, or propose alternative (server-sent events with a long-lived connection — more complex, no Google dependency).
2. **Background worker tech.** §3.2 suggests Celery; ARQ is lighter for FastAPI. Confirm the choice; default if delegated: Celery, since the user's backend already has reasonable Python ops infrastructure per the architecture doc.
3. **Gemini timeout = 90s.** Generous. Some shops use 60s. Default if delegated: 90s; revise after observing real Gemini latencies for 100 captures in production.
4. **Photo size limits.** §3.1 says ≤10 MB. Server should resize on receipt to e.g. 1600px max dimension before sending to Gemini, regardless of upload size, to control cost. Confirm or adjust default.
5. **Estimate `failed` (no items recognized) — should client offer manual entry?** When Gemini sees a photo but finds no food, the user is left with a row that has a photo but no values. Two recovery paths: (a) `Повторить` (re-estimate), (b) tap row → manually fill in values via Record screen. Default if delegated: both paths visible, manual entry as the primary CTA in this specific state.
6. **Should successful estimate trigger a haptic?** Per ADR-001 §3, no haptic on row arrival. Stay with that — don't reintroduce the buzz. Confirm.
